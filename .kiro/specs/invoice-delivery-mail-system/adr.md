# Architecture Decision Records (ADR)

## ADR-001: 部門選択機能の削減とPDF検索機能への振り替え

- **日付**: 2026-03-30
- **ステータス**: 承認済み
- **決定者**: 東海鋼材工業様・弊社合意

### コンテキスト
- 見積もりには部門選択機能（2人日）が含まれていた
- 電子帳簿保存法により、送信済み書類の検索機能（日付・金額の範囲指定、取引先名、組み合わせ検索）が法的に必要
- 現設計の処理済み一覧は過去1週間表示のみで、電帳法の要件（7年分の検索）を満たさない

### 決定
- 部門選択機能をカットし、その工数をPDF検索機能（電帳法対応）に振り替える
- 対象部門は当面1部門のみとし、部門は固定値とする

### 理由
- 他の事業部への展開は現時点で予定なし
- 電帳法対応は法的義務であり、優先度が高い
- 見積もり総額を変えずに法的要件を満たせる

### 影響
- 部門選択画面は不要
- 将来的に複数部門対応が必要になった場合は追加開発

## ADR-002: メール送信を1時間刻みのバッチ処理とする

- **日付**: 2026-03-30
- **ステータス**: 承認済み

### コンテキスト
- 当初は毎日1回（固定時刻）のバッチ送信を想定していた
- 即時送信機能の追加はアーキテクチャ変更を伴う

### 決定
- メール送信は1時間刻みのバッチ処理とする
- 即時送信機能は設けない

### 理由
- 1時間刻みであれば最長1時間待てばよく、即時送信の必要性がない
- 即時送信はアーキテクチャが複雑化する割に得られる価値が小さい

### 影響
- EventBridgeのcron式を1時間間隔に変更
- 処理予定一覧の「明日10:00処理」表記を「次回の定時処理」に変更

## ADR-003: 得意先メールアドレス・送信可否を備考2フィールドで管理

- **日付**: 2026-03-30
- **ステータス**: 承認済み

### コンテキスト
- 得意先のメールアドレスと送信可否の管理方法を決める必要があった
- 選択肢: 備考2フィールドに集約 or 基幹側に送信先マスタを新設

### 決定
- 備考2フィールドに以下を集約する
  - 請求書の送信可否（ON/OFF）
  - 納品書の送信可否（ON/OFF）
  - メールアドレス（1社1アドレスに限定）

### 理由
- 基幹側にマスタを新設するとお客様側の負担が大きい
- 備考2は既にお客様が整備予定のフィールドであり、追加コストが低い
- 1社1アドレスに限定することでデータ構造をシンプルに保てる

### 影響
- 備考2のフォーマット設計・取り決めが必要
- 複数アドレスへの送信が将来必要になった場合は再検討

## ADR-004: 電子帳簿保存法の改ざん防止措置は事務処理規程で対応

- **日付**: 2026-03-30
- **ステータス**: 承認済み

### コンテキスト
- 電帳法で改ざん防止措置が義務（事務処理規程 / システムで履歴管理 / タイムスタンプのいずれか）

### 決定
- お客様の既存の事務処理規程で対応する（システムでの履歴管理機能は追加しない）

### 理由
- お客様側に規程が既に存在する
- システムで履歴管理を実装すると追加開発コストが発生する
- S3に保存したPDFは上書き・削除しない運用とすることで、規程と実態が一致する

### 影響
- 改ざん防止のためのシステム開発は不要
- S3のバージョニング有効化を推奨（規程との整合性を強化）

## ADR-005: データ保存期間を10年とする

- **日付**: 2026-03-30
- **ステータス**: 承認済み

### コンテキスト
- 電帳法の法定保存期間は7年
- 安全策として10年の選択肢もある

### 決定
- PDF控え・データベースの保存期間を10年とする

### 理由
- 法定7年に対して余裕を持たせるお客様の判断
- 長期保存によるストレージコスト増は年間数百円程度で影響は軽微

### 影響
- S3ライフサイクルポリシーを10年に設定
- S3上のJSONファイルは10年間保持

## ADR-006: PDF生成はバッチ送信時に行い、事前確認はプレビューで対応

- **日付**: 2026-04-02
- **ステータス**: 承認済み

### コンテキスト
- 確定時にPDF生成する方式を検討したが、以下の複雑さが生じる:
  - PDF生成中/完了のステータス管理（scheduled → pdf_generating → pdf_generated）
  - 確定解除時のPDF削除・孤立問題
  - 確定→解除→再確定の繰り返しでPDFの整合性維持が困難
  - PDF生成中に確定解除された場合の書き込み先消失
  - 軽技Webから再取得後の再確定で古いPDFが無効になる問題
  - 電帳法により送信済みPDFは上書き・削除不可だが、送信前PDFとの区別が必要
- 一方で、送信前にPDFの内容を確認したいという要望が想定される

### 決定
- PDF生成はバッチ送信時に行う（確定時には生成しない）
- 送信前の確認はプレビュー機能で対応する
  - 処理予定一覧に「プレビュー」ボタンを設置
  - ボタン押下時にその1件だけPDF生成してブラウザに返す（S3には保存しない）
  - PDFのタイトルに「プレビュー」の文言を含める（正式な書類と区別）
- バッチ送信時にプレビュー文言なしの正式版PDFを生成し、送信後にS3に保存する

### 理由
- ステータスが`scheduled`と`processed`の2つだけで済む（`pdf_generating`/`pdf_generated`が不要）
- 確定・解除を何回繰り返してもPDFの管理が一切不要（JSONの作成・削除だけ）
- 送信前のPDFをS3に保存しないため、上書き・削除・孤立の問題が発生しない
- プレビューは1件ずつオンデマンド生成なので、Lambda 15分制限にも影響しない
- 確認したい人だけプレビューを使えばよく、全件のPDF事前生成が不要

### 影響
- 処理予定一覧のボタンを「ダウンロード」→「プレビュー」に変更
- PdfGenerationServiceにプレビュー生成メソッドを追加（S3保存なし、プレビュー文言付き）
- バッチ処理にPDF生成ステップを追加（送信直前に生成）
- バッチ処理時間が増加（PDF生成+送信）するが、1時間刻みで件数が分散されるため問題なし

## ADR-007: DynamoDBを廃止しS3 + Athena構成に変更（旧ADR-006）

- **日付**: 2026-03-30
- **ステータス**: 承認済み

### コンテキスト
- 電子帳簿保存法の検索要件（金額範囲・取引先名の組み合わせ検索、10年分）をDynamoDBで実現する設計を検討していた
- DynamoDBのGSIでは柔軟な複合検索（日付範囲 AND 金額範囲 AND 取引先名の部分一致）が困難
- amount_rangeのような金額帯GSIを作成する必要があり、設計が不自然になる

### 決定
- DynamoDBを完全に廃止し、S3（JSONファイル）+ Athena構成に変更する
- 処理予定/処理済みデータはS3上のJSONファイルで管理（Hiveパーティション形式）
- 通常の画面表示（処理予定・処理済み一覧）はS3 ListObjectsV2で対応
- 電帳法の検索要件のみAthena SQLで対応（日付範囲・金額範囲・取引先名のAND組み合わせ検索）
- JSONファイルをそのまま読み取り（Parquetへの変換は不要）

### 理由
- DynamoDBのGSIでは柔軟な複合検索ができない（特に金額範囲 + 取引先名の組み合わせ）
- S3 + Athenaなら標準SQLで電帳法の全検索要件を満たせる
- データ量が小さい（10年で数GB程度）ため、JSONのままAthenaで検索しても十分な性能
- DynamoDB廃止によりコスト削減（月$1未満 -> 月$0.05未満）
- アーキテクチャがシンプルになる（S3のみでデータ管理）

### 非原子性への対策

S3にはリネーム/移動APIがないため、ステータス変更は複数ステップになる。この非原子性に対して以下の対策を講じる。

**S3パス構成**:
```
documents/status=scheduled/{slip_number}.json             ← dateなし
documents/status=processed/date=YYYY-MM-DD/{slip_number}.json  ← 送信日
```

**なぜscheduledにdateを含めないか**:
- 確定日（scheduled作成時）と送信日（processed移動時）がずれうる（例: 23時に確定→翌日送信、障害で送信が遅延）
- scheduledにdateを含めると、コピー先のprocessedでdateが変わるため、パスの組み立てが複雑になる
- scheduledは件数が少なく（通常20〜30件）、ListObjectsで全件取得しても問題ないため、dateによるパーティションが不要

**なぜprocessedにdateを含めるか**:
- 処理済み一覧（過去1週間）の表示で、ListObjectsのPrefix指定による日付絞り込みが可能になる
- dateがないと10年分の全ファイルを取得する必要があり、Athenaが必須になる
- processed側のdateは送信日で確定しており、後からずれることがない

**なぜscheduled側のsent_at/processed_atで二重送信を防止するか**:
- 二重送信防止のためにprocessed側を確認する方法も検討したが、processed側にはdateがあるため、どのdateフォルダに入っているか特定できない問題がある
- scheduled側のJSON自体に`sent_at`を書き込む方式であれば、processed側を一切見に行く必要がない
- 送信ONの場合は`sent_at`で送信済みを判定（将来的にprocessed_atとタイミングが分離しても安全）
- 送信OFFの場合は`processed_at`で処理済みを判定（メール送信しないのでsent_atはnull）
- scheduled側のJSONは処理予定一覧の表示で既に読み込んでいるため、追加のAPI呼び出しも不要
- `send_result`フィールドは不要。代わりに`processed_at`と`sent_at`の2フィールドで管理:
  - `processed_at`: バッチ処理した時刻（ON/OFF両方で設定）。orphan検出にも使用
  - `sent_at`: メール送信した時刻（送信ONのみ設定）。送信済みかどうかの判定に使用
  - 現在の仕様では同一タイミングで1回のPUTで同時書き込み。将来的に送信後の追加処理が発生した場合に分離可能な設計

**バッチ送信の手順**:
1. PDF生成（正式版、S3に保存）
2. メール送信を実行（送信ONのみ。送信OFFはスキップ）
3. scheduled側のJSONに`processed_at`（と送信ONの場合は`sent_at`）を書き込む（PUT上書き）— これが処理済みの証拠
4. processed側にコピー
5. scheduled側を削除

**二重メール送信の防止**:
- 次回バッチでscheduled側のJSONを読み、`processed_at`が既に設定されていれば処理済みと判断
- 送信をスキップし、コピー+削除のリカバリのみ実行
- processed側を検索する必要はない（scheduledのJSON内で完結）

**障害シナリオと対策**:

| クラッシュタイミング | scheduled | processed | 次回バッチの動作 |
|---|---|---|---|
| 1の後（PDF生成後、送信前） | sent_at/processed_atなし | なし | PDF再生成→再送信 |
| 2の後（送信後、書き込み前） | sent_at/processed_atなし | なし | PDF再生成→再送信（※） |
| 3の後（書き込み後、コピー前） | sent_at/processed_atあり | なし | 送信スキップ、コピー+削除 |
| 4の後（コピー後、削除前） | sent_at/processed_atあり | あり | 送信スキップ、削除のみ |

※メール送信直後・JSON書き込み前のクラッシュによる再送信は、どのデータストア（DynamoDB、RDS等）を使っても原理的に防げない（分散トランザクションの限界）。発生確率は極めて低く、許容する。
※PDF再生成は冪等（同じデータから同じPDFが生成される）なので問題ない。

**画面表示のルール**:
- 同じ伝票番号がscheduledとprocessedの両方に存在する場合、processedを優先して表示する
- これにより、削除失敗で両方に残っていても、ユーザーには正しい状態が見える

### 影響
- DocumentRepositoryクラスを廃止し、S3ベースのDocumentStorageに置き換え
- DynamoDBテーブル定義、GSI設計はすべて削除
- S3ファイル構成:
  - `documents/status=scheduled/{slip_number}.json`（dateなし）
  - `documents/status=processed/date=YYYY-MM-DD/{slip_number}.json`（送信日）
- 処理予定一覧: ListObjectsで全件取得（件数少ない）
- 処理済み一覧（過去1週間）: ListObjectsで日付プレフィックス指定（7回）
- 電帳法検索（10年分、複合条件）: Athena
- Glue Data CatalogにAthena用テーブル定義が必要

## ADR-008: バッチ送信を親子Lambda構成にする（1件1Lambda）

- **日付**: 2026-04-02
- **ステータス**: 承認済み

### コンテキスト
- バッチ処理でPDF生成+メール送信を行う設計（ADR-006）により、1回のバッチ処理時間が増加
- Lambda実行時間の上限は15分
- 月末ピーク時は1時間あたり数十件の処理が発生する可能性がある
- 1件あたりPDF生成5秒+送信数秒で、大量件数を1つのLambdaで処理するとタイムアウトのリスクがある

### 決定
- バッチ処理を親Lambda（オーケストレーター）と子Lambda（1件処理）に分離する
- 親Lambda: EventBridgeトリガー（1時間ごと）で起動。`scheduled/`をListObjectsし、1件ずつ子Lambdaを非同期Invokeする
- 子Lambda: 1件の書類に対してPDF生成→送信→processed_at書き込み→processedにコピー→scheduled削除を実行

### 理由
- 1件あたり数十秒で完了するため、15分制限に余裕がある
- 1件失敗しても他の書類の処理に影響しない（障害の局所化）
- 失敗した書類はscheduledに残り、次回バッチで自動リトライされる
- 処理ロジックが1件分だけなのでシンプルでテストしやすい
- コスト増は月末ピーク250件でも$0.00005程度で実質ゼロ

### 影響
- Lambda関数が2つ必要（親: batch-orchestrator、子: batch-process-single）
- 親Lambdaは軽量（ListObjects + Invoke）なので数秒で完了
- SendGridレート制限対策: 親Lambdaで100ms間隔でInvokeするか、子Lambda内でリトライ
- SAMテンプレートにLambda関数を1つ追加
